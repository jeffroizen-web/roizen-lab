# Content Contract — config-for-one (templatize-content increment 1)

**Status:** shipped 2026-06-27 (Ace Scout, background-dispatch #2 `templatize-content`, Kleiber MSG-1b0111).
**Files:** `content/site-content.json` (the contract) · `tests/test_content_contract.py` (the drift guard).

## What this is

The first, autonomous-safe step of templatizing the Roizen Lab site: the
Jeff-specific, **stable** rendered-site content is extracted out of the
monolithic `compare-purple-gold.html` into one reviewable/editable data
contract. This is the "config" half of config-for-one (single-PI, no
multi-tenancy assumed).

The canonical HTML is **still authoritative for RENDER** — nothing reads the
JSON to build the page yet. So this increment changes nothing the visitor sees
and cannot endanger the launch-ready artifact. The value delivered now:

1. **One place to see/edit all site content** (meta, hero, the 7 Big
   Questions + figures, people, news, donate, links).
2. **A drift guard** (`tests/test_content_contract.py`): every value in the
   contract must still appear in the live HTML, so the two cannot silently
   diverge. Normalization (entity-unescape + tag-strip + whitespace-collapse)
   makes the match robust to inline markup.

## Schema (top-level keys)

`meta` (title/description/canonical/OG/twitter) · `hero` · `big_questions`
(`questions[7]`: id, title, body, arc_anchor, figure{src|placeholder}) ·
`team` · `collaborators` · `peers` · `mentors` · `publications` (refs
`data/publications.json`) · `news` · `alumni` · `donate` · `contact` ·
`links`. A leading `_meta` block documents provenance and is excluded from the
drift guard.

## Deliberate exclusions (Producer-Owns)

The **`.field-of-interest` spans** in the collaborator/peer/mentor cards are
generated by `scripts/wire_letter_writers.py` from `docs/letter_writers.json`
on a cron cadence. Per-person **shared-publication lists** are similarly
volatile. These are NOT in the contract — tracking them would make the drift
guard flap on every cron run, and they are owned by the letter-writer
pipeline, not this contract. `test_no_cron_owned_fields` enforces that the
contract never starts tracking them.

The hidden `merch` section (commented out for launch) is recorded in
`_meta.hidden_sections` but not carried as active content.

## Next increment — JEFF-GATED (not autonomous)

Flipping the canonical page to **render from this contract** requires either a
build step (generator: contract + template → static HTML) or client-side
rendering. Both cross the documented **no-build-system** constraint
("Plain HTML/CSS/JS, edit files directly" — CLAUDE.md Key Constraints), and
client-side rendering would also harm SEO (the JSON-LD/meta/content must be in
the served HTML for crawlers). That is a direction call for Jeff, not a
reversible autonomous build. Recommended shape when greenlit: a small Python
generator producing the static HTML at build time (SEO preserved), gated by
the existing `tests/site-qa.spec.js` suite + a content-equivalence check
against the current page before any flip.
